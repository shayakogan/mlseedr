# Seedr — Маркетинговый ML и движок удержания — Отчёт по проекту

Дата: 2026-06-22. Полный отчёт: зачем, что построили, новые фичи, польза для бизнеса,
архитектура, как выросли показатели и что осталось. Все данные в ClickHouse `ml.*`;
код+доки в GitHub `mlseedr`; файлы продублированы на `stat2.seedr.cc:/root/mlseedr`.

---

## 1. Зачем мы это делали
Seedr — freemium облачный torrent/download/стриминг: ~70K активных в неделю, но лишь
~3.2–3.5K платящих и конверсия free→paid на письмо ~0.06%. Маркетинг был «по всем подряд».
**Цель:** превратить телеметрию ClickHouse в управляемый данными маркетинг — знать, *кто*
конвертнётся, уйдёт или ценен, и *действовать* (апселл / удержание / возврат) на нужных
людей с нужным сообщением.

---

## 2. Что построили (результаты)
- **Верифицированный research** (прогоны по 106 агентов): сегментация freemium + какие
  сигналы предсказывают конверсию — основа всех моделей и сегментов.
- **8 маркетинговых сегментов** (win-back, soft-cancel, storage-pressure, heavy-bandwidth,
  стримеры, брошенная корзина, dormant payers, monthly-loyal) — посчитаны вживую, выгружены.
- **3 основные модели** (датасеты + обучение):
  - Конверсия («оплатит за 14д после письма»), 7.17M строк.
  - Отток («активный подписчик отменит за 30д»), 17.5K строк.
  - LTV / CLV («выручка за следующие 365д»), 647K строк.
- **Retention priority** = ценность × риск → `ml.retention_priority`.
- **Win-back** 591 уже ушедших (30д), сгруппированы по персонам.
- **Content-affinity** — совершенно новый источник (admin FS API Seedr): метаданные
  хранилища/файлов → персоны, 42 фичи → `ml.user_content`.
- **Edge/QoS, task-usage, billing-health** — фича-таблицы из ClickHouse.
- **Customer-360** — одна строка на юзера, объединяет всё + `next_best_action`.
- **Live-снимки** (`customer_360_history`), **uplift-holdout** фреймворк, **аудитории кампаний**.

## 3. Новые собранные фичи (сигнал, которого раньше не было)
| Группа | Источник | Примеры |
|---|---|---|
| Интент | телеметрия | `promo_sub` (visited-subscription, ×7 raw lift), воронка pricing/goal |
| Тайминг биллинга | метаданные Partytime | **`expires_on`** (точная дата списания), `has_payment_method` (120/120 истёкших — без карты) |
| Контент-аффинити | **FS API (новое)** | `content_persona` (video_streamer 60% / empty 24% / …), storage_gb, доли video/audio/ebook, `n_lost_files`, `files_added_30d`, `saw_walkthrough`, `last_signin_day`, флаги привлечения |

| Использование | task.* | downloads_30d, **task_failure_rate** (фрустрация) |

| Динамика вовлечённости | производные | RFM (recency/frequency/monetary), тренд трат, тренды активности/сессий |

## 4. Как это помогает бизнесу
- **Точечные кампании вместо рассылок «всем»:** `next_best_action` на юзера —
  HD-апселл (1.1K активных стримеров, упирающихся в 429/столлы), реактивация-empty (3.0K
  платят, но пусто), fix-payment (388 сбоев оплаты), VIP-nurture (1.1K, LTV $113), urgent-save (71).
- **Тратить там, где деньги:** 44% платящих держат **83%** будущей ценности → фокус туда.
- **Win-back** на $110K пожизненной выручки уже ушедших, персонализировано по персонам.
- **Дёшево ловить involuntary-отток:** 220 юзеров почти гарантированно уйдут из-за
  отсутствия карты → «обнови карту», без скидки.
- **Мерить причинность, а не корреляцию:** рандомизированный holdout → истинный uplift письма.

## 5. Архитектура
```
ИСТОЧНИКИ                       ФИЧА-ТАБЛИЦЫ (ml.*)               МОДЕЛИ / СКОРИНГИ           ДЕЙСТВИЕ
ClickHouse seedr_telemetry ─┐   user_content (контент)           conversion (LogReg/NN)     customer_360
  (web/email/subs/tasks)    ├─► user_edge   (edge/QoS)      ┐     churn (GBM)          ┐    + next_best_action
revenue_facts (10 лет)      ├─► user_tasks  (usage)         ├──► LTV two-part (P×E)     ├──► аудитории кампаний
payments DB (Partytime)     ├─► user_billing_health         │     retention_priority    │    + holdout (uplift)
Seedr FS API (НОВОЕ)        ┘   ltv_scores / churn_scores   ┘     (ценность × риск)      ┘    customer_360_history
                                                                                              (дневные live-снимки)
```
- **Модели:** градиентный бустинг (HistGradientBoosting) везде; LTV — two-part hurdle
  (P(оплатит) × E(сумма)); опционально multi-task нейро-backbone со сменными головами.
- **Идентичность:** join по `user_id`. **Хранение:** всё в ClickHouse `ml.*` (есть запись);
  PII никогда не в git. **Обновление:** `snapshot_daily.sh` пересобирает CH-фичи + дописывает
  датированный снимок `customer_360`.

## 6. Как выросли показатели
| Модель | Метрика | Результат |
|---|---|---|
| **Конверсия** | ROC-AUC / lift@1% | **0.95 / ×65** — топ-1% по скору платят ~3.7% против базы 0.056% |
| **Отток** | ROC-AUC | **0.60 → 0.78** после биллинг-фич; precision топ-10% **67%**, lift ×3.8 |
| **LTV** | P(pay) AUC; ранжир. | AUC **0.91**; two-part бьёт persistence (MAE −12%, RMSE −22%); топ-10% ловят **39%** выручки года |
| **Retention priority** | концентрация | P1+P2 = 44% платящих держат **83%** будущей ценности |
| **Контент-лифт** | churn / LTV | +0.01 AUC / +0.016 Spearman (на ретро-моделях скромно; реальный лифт ждём на live-снимках) |
| **Калибровка** | ECE | 0.174 → **0.0004** (изотоническая) — вероятности пригодны для €-решений |

Честные оговорки: конверсия на потолке ~0.95 (определяется прошлым платёжным поведением);
точный сигнал даты списания (`expires_on`) только с июня 2026 → улучшает *текущий* скоринг,
но поднимет *обученный* AUC лишь после переобучения на накопленных данных; контент-лифт на
ретро-моделях мал, т.к. это снимок «сейчас» против исторических меток.

## 7. Что осталось
1. **Запустить кампанию на `arm='treatment'`** (holdout уже назначен) → снять первый
   реальный **uplift** — единственный нерешённый причинный вопрос.
2. **Переобучить churn/LTV на ~30 днях live-снимков** (`customer_360_history`) → реальный
   (не ретро) лифт от content/edge/billing; ожидаемый churn AUC ~0.85.
3. **storage_used_pct** — нужна квота хранилища (из `billing_plan_id`→тариф или
   `/dynamic/get_space`); единственный пробел контента.
4. **Расписание** `snapshot_daily.sh` на хосте с персистентным туннелем (autossh) — crontab
   в рабочей среде недоступен.
5. **Карта `billing_plan_id` → тариф/цикл** (особенно 1000-серия) от биллинг-команды.
6. Опционально: гео/PPP-анализ цен; контент-aware креативы; bandit/RL-таргетинг после
   накопления uplift-данных.
